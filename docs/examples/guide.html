Lumen v0.2 — architecture overview

guide example · @the-forge-flow/lumen · 2026-04-26
Glance

A skill bundle that ships the renderer next to the recipe.

Lumen is a Claude Code plugin and PI extension. It ships eight skills for generating single-file HTML visuals: diagrams, charts, mermaid, slides, galleries, guides, recaps, fact-checks. Three of those skills also have a deterministic renderer wired through the PI lumen-generate_visual tool; the rest run as LLM-authored skills in Claude Code.

Request flow
Step by step
1
Tool call lands
A PI session calls lumen-generate_visual with { type, content, title, aesthetic?, theme?, filename? }. The type is the discriminator that selects a renderer.
2
Validate
validateParams(rawParams) checks the typebox schema and rejects unknown types. content's shape depends on type: a string for mermaid, a structured object for diagram and chart.
3
Route
A small switch in generateVisual() picks the renderer: renderMermaid for the 5 mermaid types, renderDiagram for fgraph, renderChart for charts. Anything else throws NotImplementedError.
4
Render
Each renderer returns a complete self-contained HTML string. No CDN deps for offline-safe playback.
5
Write + open
The HTML is written to a tracked recent-files list, then handed to openInBrowser via the PI extension API.
Recent design choices
Why ship a deterministic renderer at all?

The earliest skills were entirely LLM-authored: Claude composed the HTML, including all diagram drawing, every time. That works but spends tokens on exact layout math the renderer can do for free, and it makes the output non-reproducible.

For diagram and chart shapes that have well-defined geometry — sequence diagrams, bar charts, layered topologies — a deterministic renderer takes a structured input and emits the same HTML every time. Tokens get spent on composition (what the diagram should show) rather than rendering (where each pixel goes).

Skills that need real compositional judgment — slides, guides, recaps, fact-checks — stay LLM-authored.

Why one PI tool, many types?

Earlier sketches had one PI tool per visual kind (lumen-make-chart, lumen-make-diagram, …). That bloats the tool surface PI shows the model and forces every new visual type to ship as a new tool.

The current shape is one tool, lumen-generate_visual, with a type discriminator. New visual types extend the schema's enum and add a renderer; the tool itself doesn't change. The router lives in src/index.ts at generateVisual().

Why promote aesthetics to skills/_shared/?

v0.1.x had each skill carry its own copy of the palette CSS. When a tweak to the editorial palette was needed, it had to be made in five places, and the five copies drifted.

v0.2 promoted aesthetics to skills/_shared/aesthetics/ — one CSS file per palette, consumed by every skill. The skill-presence loop in CI explicitly skips _shared so the directory doesn't have to pretend to be a skill.

Why migrate from @sinclair/typebox to unscoped typebox?

Not lumen's choice — pi-coding-agent / pi-ai / pi-tui jumped from 0.66 → 0.70 and replaced their @sinclair/typebox 0.34 peer dep with the unscoped typebox 1.x package (same library, republished under a new name with a major bump).

Bumping our pi peers to ^0.70.2 meant matching the new typebox name and version. Our usage in src/index.ts only touches Type.{Object,String,Array,Record,Unknown,Union,Optional} — all unchanged across the rename — so the diff was three lines.
