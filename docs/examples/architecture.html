Lumen v0.1.4 — architecture

composite output · diagram → guide → fact-check · 2026-04-26
Glance

A skill bundle that ships the renderer next to the recipe.

Lumen is a single repository that publishes as both a Claude Code plugin and a PI coding-agent extension. It ships eight atomic skills for single-file HTML visuals — diagrams, charts, mermaid, slides, galleries, guides, recaps, fact-checks — plus one composite that orchestrates three of them into this very document.

Three skills (lumen-mermaid, lumen-diagram, lumen-chart) also expose a deterministic TypeScript renderer through the PI tool lumen-generate_visual; the rest are LLM-authored on the Claude Code path.

01 — Skill tiers

Capabilities, composites, playbooks

Lumen organises skills into three tiers inspired by Skill Graphs 2.0: capabilities are atomic and never invoke each other; composites orchestrate 2–10 capabilities in a fixed pipeline; playbooks are human-driven orchestrators over composites. Today the tree has eight capabilities (skills/), one composite (composites/lumen-architecture-doc/), and zero playbooks — the playbooks/ directory ships empty in v0.1.x by design.

The plugin manifest at .claude-plugin/plugin.json registers all three directories as skill roots, and the PI export in package.json#pi.skills mirrors them under dist/ via the prebuild script.

02 — PI request flow

One tool call, five stages

The PI extension exposes a single tool, lumen-generate_visual, defined in src/index.ts. Every call passes through the same five stages: validation against the typebox schema, routing on the type discriminator, rendering to a complete HTML string, writing to a tracked recent-files list, and opening in the user's default browser.

This shape lets new visual types extend the schema enum and add a renderer function without changing the tool surface PI presents to the model.

03 — Renderer routes

Three render paths, one rejection

The router in generateVisual() dispatches on type to one of three renderer functions: renderMermaid handles the five mermaid types (flowchart, sequence, er, state, mermaid_custom), renderDiagram handles four fgraph topologies (sequence, layered, linear-flow, radial-hub), and renderChart handles four chart types (bar, pie, line, table).

Every other declared VisualType — architecture, slides, table, diff, plan, timeline, dashboard — currently throws NotImplementedError. The rejection path is a feature: it tells the calling agent that the matching lumen-* skill is the right invocation surface for those types until the renderer lands.

04 — Dual distribution

Same skills, two consumers

Both distributions resolve to the same skill tree. The Claude Code plugin manifest (.claude-plugin/plugin.json) declares "skills": ["./skills/", "./composites/", "./playbooks/"]; the PI extension entry (package.json#pi) declares the same three directories under their dist/ mirrors plus the compiled ./dist/index.js as the extension module. The prebuild script copies the skill trees into dist/ so a single npm publish ships both.

Only PI receives the deterministic TypeScript renderer; Claude Code consumes skills as authoring instructions only.

Cross-subsystem flow A — generate a diagram from PI

When a PI session asks for a fgraph diagram, the call enters the request-flow subsystem above and routes into renderDiagram, which loads an aesthetic from skills/_shared/aesthetics/ via loadAestheticCss(), dispatches on the topology field to one of four topology renderers, wraps the body in the standard fgraph document shell, and returns a complete HTML string.

Cross-subsystem flow B — produce this very document

The lumen-architecture-doc composite is the orchestrator behind this file. It runs a fixed three-step pipeline with no agent discretion over the order; if any step fails the composite surfaces the failing capability rather than shipping a partial document.

  Step 1: lumen-diagram   × N subsystems → temp HTML files
  Step 2: lumen-guide     (single-file mode) → embed Step 1 SVG/diagrams
  Step 3: lumen-fact-check → corrections + verification summary tab
Cross-subsystem flow C — release

Versioning is fully driven by release-please. The x-release-please-version markers in src/index.ts (LUMEN_VERSION), .claude-plugin/plugin.json, every SKILL.md frontmatter, and README.md status line are kept in sync by release-please-config.json. No file ever has its version edited by hand.

Architectural decisions
Why ship a deterministic renderer at all?

The earliest skills were entirely LLM-authored: Claude composed the HTML, including all diagram drawing, every time. That works but spends tokens on exact layout math the renderer can do for free, and it makes the output non-reproducible.

For diagram and chart shapes that have well-defined geometry — sequence diagrams, bar charts, layered topologies — a deterministic renderer takes a structured input and emits the same HTML every time. Tokens get spent on composition (what the diagram should show) rather than rendering (where each pixel goes).

Skills that need real compositional judgment — slides, guides, recaps, fact-checks — stay LLM-authored.

Why one PI tool, many types?

Earlier sketches had one PI tool per visual kind (lumen-make-chart, lumen-make-diagram, …). That bloats the tool surface PI shows the model and forces every new visual type to ship as a new tool.

The current shape is one tool, lumen-generate_visual, with a type discriminator. New visual types extend the schema's enum and add a renderer; the tool itself doesn't change. The router lives in src/index.ts at generateVisual().

Why three tiers (capability / composite / playbook) instead of flat skills?

Atomic skills with no cross-skill calls are easy to test, easy to invoke, and easy to evolve independently. But real workflows — "produce an architecture doc" — need orchestration.

Pushing orchestration into a separate composite tier with a fixed pipeline keeps atomics atomic, and gives the agent a deterministic recipe rather than open-ended discretion over which skills to call. Playbooks (compound tier) sit above composites for human-driven multi-composite workflows, and ship empty in v0.1 because no real playbook has earned its keep yet.

Why dogfood the composite for this very document?

The composite's reliability contract — fixed pipeline, no skipping steps, fact-check before shipping — is meant to produce documents that survive review. Using it for lumen's own architecture doc tests the contract against a target (this codebase) where every claim can be verified locally; any drift between the rendered doc and the code shows up immediately in the fact-check tab. If the composite can't produce its own architecture doc accurately, that's a bug in the composite, not a documentation problem.

Composite-spec deviations (transparent)
low
The composite spec calls for embedding subsystem diagrams by extracting their <svg> block and inlining it. The fgraph topologies render as HTML divs layered with an SVG edge overlay, not pure SVG, and the four topology CSS sections collide on .fgraph-wrap's aspect ratio. We embedded each subsystem diagram as an <iframe srcdoc> instead — preserves single-file output, eliminates CSS contamination, and keeps every diagram exactly as the renderer emitted it. Spec amendment candidate for v0.2.
